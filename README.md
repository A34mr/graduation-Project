**Dent AI** is a state-of-the-art, end-to-end web ecosystem designed to bridge the gap between dental clinics, professional dentists, and patients. By integrating an **AI-driven OPG radiographic cavity detection model**, real-time secure messaging, structured clinic profiles, slot-based booking systems, and clinical report engines, Dent AI streamlines dental healthcare operations.

---

## 🌟 Core Features

### 👤 1. For Patients
- **Smart Geolocation Search**: Locate nearest dental clinics using latitude/longitude geo-coordinates, with filtering options for ratings, price range, city, and specialties.
- **Appointment Booking**: Find and reserve available slots for specific doctors, view history, and process payments (mock checkout integration).
- **AI Radiograph Diagnosis**: Upload dental X-rays/OPG scans for instant AI cavity detection.
- **Download Medical Reports**: Access doctor-finalized treatment plans, medication prescriptions, cost estimates, and download them as high-quality PDF files.
- **Real-Time Communication**: Chat with doctors instantly using live chat.

### 🥼 2. For Doctors
- **Interactive Dashboard**: Track daily appointments, active patients, and clinical statuses.
- **X-Ray Review & Review Overrides**: Inspect patient-uploaded scans, view AI confidence levels, override/adjust diagnoses, and add clinical notes.
- **Treatment & Report Generator**: Generate comprehensive treatment plans with medication dosages, durations, cost breakdowns, and finalize them for PDF generation.
- **Real-Time Patient Consultations**: Converse with patients to provide virtual consultations.

### 🏢 3. For Clinic Administrators
- **Clinic Profiling**: Manage clinic addresses, contact details, description, services, and locations.
- **Doctor Management**: Register new doctors, manage their credentials, years of experience, and specialty profiles.
- **Slot Allocation**: Automatically or manually schedule appointment availability slots.

### 🔑 4. For System Administrators
- **Platform Supervision**: Deactivate/activate accounts, approve newly registered clinics, and oversee ecosystem usage stats.

## 📊 Database Schemas

The MongoDB database models are structured as follows:
- **`User`**: Core user profiles storing emails, hashed passwords, avatars, active statuses, and roles (`patient`, `doctor`, `clinic_admin`, `admin`).
- **`PatientProfile`**: Extends the user model for patients to store DOB, gender, chronic illnesses, allergies, and patient-specific medical history.
- **`DoctorProfile`**: Extends user profiles for doctors, linking them to a `Clinic` and storing license numbers, specialties, experience, and verification state.
- **`Clinic`**: Details on dental clinics, including geo-coordinates (2dsphere indexes), verified status, list of affiliated doctors, rating stats, and working hours.
- **`Appointment`**: Links patient, doctor, and clinic, storing schedules, status (`pending`, `confirmed`, `completed`, `cancelled`), and payment statuses.
- **`Slot`**: Time slots generated by doctors for patients to choose and book.
- **`Image`**: Stores uploaded OPG scans, Hugging Face AI results (cavity locations, severity, confidence), and doctor review overrides.
- **`Report`**: Detailed clinic report containing the final diagnosis, treatment plans (procedure name, details), prescriptions (drug names, frequency, duration), billing estimates, and AI findings.
- **`ChatMessage`**: Persists conversations, sender/receiver references, and read timestamps.
- **`Review` / `Complaint`**: Quality assurance models for clinic ratings and administrative alerts.

## 🔒 Security & Performance Features
1. **Short-Lived Access Tokens**: Employs short-lived JWT access tokens (`15m` expiry) coupled with long-lived refresh tokens (`7d` expiry) stored securely.
2. **Token Blacklist**: Denies reuse of logged-out or refreshed tokens using an in-memory blacklisting store.
3. **Rate Limiting**: Protects system endpoints using `express-rate-limit` (e.g., maximum 50 login attempts and 20 registration attempts per hour).
4. **CORS Isolation**: Strict, controlled CORS policies using verified origins instead of wildcard entries, with explicit custom error handlers.
5. **Database Sanitization**: Prevents credential leaks by enforcing strict model select fields and avoiding logging of raw database URIs.

## 🚀 Installation & Setup

### Prerequisites
- [Node.js](https://nodejs.org/) (v18.0.0 or higher recommended)
- [MongoDB](https://www.mongodb.com/try/download/community) (running locally on port 27017 or a MongoDB Atlas URI)

### 1. Clone & Initialize the Project
```bash
git clone <repository-url>
cd graduation-Project
```

### 2. Configure Environment Variables

#### Backend Configuration (`backend/.env`)
Create a `.env` file inside the `backend` folder and populate it:
```env
# Server Configuration
PORT=5000
NODE_ENV=development

# MongoDB URI (Update with your credentials if using Atlas)
MONGO_URI=mongodb://127.0.0.1:27017/dentai

# JWT Cryptographic Secrets (Generate strong keys for production)
JWT_SECRET=YOUR_CRYPTO_SECURE_JWT_SECRET_STRING
JWT_REFRESH_SECRET=YOUR_CRYPTO_SECURE_JWT_REFRESH_SECRET_STRING

# Frontend URL
FRONTEND_URL=http://localhost:3000
```

#### Frontend Configuration (`frontend/.env`)
Create a `.env` file inside the `frontend` folder:
```env
VITE_BACKEND_URL=http://localhost:5000
```

## 💻 Running the Application

### Option A: Running Backend
1. Navigate to the backend directory:
   ```bash
   cd backend
   ```
2. Install server dependencies:
   ```bash
   npm install
   ```
3. Run the development server (runs with `nodemon` for hot reloading):
   ```bash
   npm run dev
   ```
   *Note: On startup, MongoDB will automatically connect and seed mock clinics, doctors, patient users, reports, and slots.*

### Option B: Running Frontend
1. Open a new terminal and navigate to the frontend directory:
   ```bash
   cd frontend
   ```
2. Install client-side dependencies:
   ```bash
   npm install
   ```
3. Launch the Vite development server:
   ```bash
   npm run dev
   ```
4. Access the web interface in your browser (usually at `http://localhost:5173`).

## 💬 Real-Time Chat System Architecture

The chat platform utilizes **Socket.IO** to build secure, bidirectional connections.

1. **Authentication Gate**: Sockets must supply a valid JWT access token during handshakes (`socket.handshake.auth.token`).
2. **Room Isolation**: Private chats utilize isolated room channels. The room names are generated dynamically using sorted user IDs:
   ```javascript
   const roomName = [userId, otherUserId].sort().join('_');
   ```
3. **Core Events**:
   - `joinChat`: Directs the socket to enter its private conversation room.
   - `sendMessage` / `newMessage`: Dispatches messages securely, saving them to MongoDB concurrently.
   - `typing`: Notifies users when their conversational partner is typing.

## 🔬 AI Diagnosis Pipeline

```
[Patient uploads scan via App] ➔ [Saved locally via Multer] 
                                  ➔ [Sent to Hugging Face API] 
                                    ➔ [Cavities Detected & Confidence Analyzed] 
                                      ➔ [Saved to DB / Fallback to Mock if offline]
```
- **Hugging Face Model Space**: Evaluates scans for cavities, returns locations, confidence scores, and determines severity levels (`mild`, `moderate`, `severe`).
- **Resilience Fallback**: If the Hugging Face API returns a 404 or connection failure, the system falls back to a development mock analysis to guarantee smooth testing workflows.
